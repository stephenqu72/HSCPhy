# Local Ollama Gemma 4 Provider Design

Date: 2026-06-19

## Goal

Add `Local Ollama - gemma4:26b` as an LLM provider that supports every existing LLM-powered workflow when the Streamlit app is running on the user's local desktop.

## Scope

The Ollama provider will support:

- Text and graph answer generation and regeneration
- Question type classification
- Chat about the current question
- Student-answer feedback
- Flash-card generation
- Bulk answer generation
- Question-image input for multimodal prompts

The feature is intended for localhost use only. Existing Gemini behavior, API-key selection, permissions, shared teacher-answer storage, and answer-cache formats will remain unchanged.

## Architecture

Introduce a small provider abstraction responsible for text and image requests. Existing feature code will call this shared interface instead of constructing Gemini models directly.

The Gemini implementation will preserve current behavior. The Ollama implementation will call the native local HTTP API at `OLLAMA_HOST`, defaulting to `http://localhost:11434`, and use model `gemma4:26b` by default. Configuration may be overridden with `OLLAMA_HOST` and `OLLAMA_MODEL` environment variables.

The selector will expose a clearly labelled `Local Ollama - gemma4:26b` option. Provider selection will remain in Streamlit session state like the current model selection.

## Request Flow

For text-only operations, the provider sends the existing prompt as a chat message. For image operations, it converts the PIL question image to PNG bytes, encodes it as base64, and includes it in the Ollama message's `images` field.

The provider returns a normalized response object or plain response text so downstream parsing, classification, plot extraction, feedback, flash cards, and answer saving continue to use the current code paths.

Ollama requests are sequential. This preserves the existing bulk-generation requirement that one answer must finish and be stored before processing the next question.

## Errors

Connection failures will show a concise instruction to start Ollama. A missing model response will tell the user to run:

```powershell
ollama pull gemma4:26b
```

HTTP errors, timeouts, malformed responses, and empty generations will be handled without overwriting a previously saved answer. Errors in bulk generation will identify the affected question and allow the existing progress workflow to continue or stop consistently with current behavior.

## Compatibility

No new Python dependency is required; the implementation will use Python's standard HTTP facilities. The local Ollama option will not be selected automatically on Streamlit Cloud. Gemini key configuration will only be required when a Gemini provider is selected, allowing a localhost Ollama-only session to start without a Gemini key.

Saved answers and flash cards remain provider-independent. A result generated by Ollama uses the same unique question key and can be reused in either practice mode by all users according to the existing authorization rules.

## Testing

Tests will cover:

- Provider-option identification and routing
- Text-only Ollama request construction
- Base64 image request construction
- Normalized Ollama response handling
- Connection, missing-model, HTTP, malformed-response, and empty-response errors
- Gemini routing regression coverage
- Existing answer-cache behavior remaining provider-independent

Implementation will follow test-driven development: each provider behavior will first be represented by a failing test, followed by the minimum production change required to pass.
